### 버그 5: Ingest 추출 무응답 (Ollama 타임아웃 및 대용량 PDF 처리 실패)

**현상**

파일 업로드 후 추출 버튼을 누르면 "소스 파일 읽는 중..." 메시지 이후 SSE keepalive만 반복되고 결과가 나타나지 않음. 잡 패널에서 오류 발생 시에도 상태가 "완료 ✓"로 표시되어 실패 여부를 알 수 없었음.

**원인**

1. **Ollama 타임아웃**: litellm 기본 타임아웃(600초)이 `gemma4:31b` 같은 대형 모델의 실제 추론 시간(단순 응답에도 362초 소요 실측)보다 짧아 `litellm.Timeout` 예외 발생.
2. **대용량 PDF 컨텍스트 초과**: `Towards an AI co-scientist.pdf`(308,906자, ~77K 토큰)처럼 PDF 전문을 그대로 전송하면 모델 컨텍스트 한계를 초과하여 타임아웃 또는 OOM 유발.
3. **`complete()` 상태 덮어쓰기**: `emit_error()`로 `status = "failed"`가 설정된 후 `finally` 블록의 `complete()`가 `status = "done"`으로 덮어써 잡 패널에서 실패 작업도 항상 "완료 ✓"로 표시됨.

**해결**

1. **타임아웃 연장** (`wiki_cli/llm.py`): litellm 호출에 `timeout=_LLM_TIMEOUT` 추가. 기본값 1200초(20분). `WIKI_LLM_TIMEOUT` 환경변수로 조정 가능.
2. **PDF 텍스트 자동 트런케이트** (`wiki_cli/llm.py`): `call_with_file()`에서 텍스트가 `_PDF_MAX_CHARS`(기본 60,000자) 초과 시 앞부분만 사용하고 생략 안내 문자열 삽입. `WIKI_PDF_MAX_CHARS` 환경변수로 조정 가능.
3. **`complete()` 상태 보존 수정** (`wiki_web/progress.py`): `status != "failed"`일 때만 `"done"`으로 변경하도록 조건 추가.

**수정 파일**: `wiki_cli/llm.py`, `wiki_web/progress.py`

### 버그 4: 대시보드 첫 페이지 500 Internal Server Error

**현상**

서버 시작 후 `/` 접속 시 즉시 500 Internal Server Error 발생.

**원인**

`wiki_web/routers/wiki.py`의 `dashboard` 함수에서 `active_domain["wiki_root"]`를 직접 참조함. 개선 7에서 도메인 구조가 `{id, name, wiki_root}` → `{id, name, folder}`로 마이그레이션되었으나 해당 라우터만 업데이트되지 않아 `KeyError: 'wiki_root'` 발생.

**해결**

`Path(active_domain["wiki_root"])` 직접 참조를 `cfg.get_wiki_root(c)` 헬퍼 호출로 교체. 다른 라우터들(`documents.py`, `query.py`, `synthesis.py`, `lint.py`)은 이미 동일 헬퍼를 사용하고 있었음.

**수정 파일**: `wiki_web/routers/wiki.py`

### 버그 1: `ProgressChannel = None` 설정으로 인한 TypeError 발생

**현상**

질문을 제출하면 아무런 응답 없이 브라우저 개발자 도구의 네트워크 탭에 500 Internal Server Error가 발생함. 서버 로그에는 `TypeError: 'NoneType' object is not callable` 오류가 기록됨.

**원인**

`progress.py`의 126번째 줄에 `ProgressChannel = None`으로 하위 호환성 코드가 작성되어 있었음. `query.py`는 `from wiki_web.progress import ProgressChannel`을 통해 채널 클래스를 임포트하려 했지만, 실제로는 `None` 객체를 가져오게 됨. 이로 인해 `submit_query` 함수 내에서 `channel = ProgressChannel()` 호출 시 `NoneType` 인스턴스화 시도로 에러 발생.

**해결**

`ProgressChannel = None`을 삭제하고, query 라우터 전용으로 경량화된 실제 `ProgressChannel` 클래스를 구현. `emit`, `emit_error`, `close`, `stream` 인터페이스를 제공하여 SSE 통신이 정상적으로 이루어지도록 수정함.

**수정 파일**: `wiki_web/progress.py`

### 버그 2: 답변(HTML)이 SSE done 이벤트로 전달되지 않음

**현상**

채널 에러가 해결된 후에도, "검색 중..." 진행 과정(SSE chunk)은 표시되지만 최종 답변이 렌더링되지 않음. `job_id/stream` 통신이 끝나면 화면에 빈 공간만 남음.

**원인**

`submit_query` 라우터의 백그라운드 태스크 `_run()`은 `answer_holder` (리스트) 클로저에 답변을 저장함. 하지만 클라이언트가 SSE에 연결하는 `query_stream` 라우터에서는 `answer_holder`에 접근할 방법이 없었음. (기존 코드에서는 `try ... pass`로 처리되어 답변을 전혀 가져오지 못함) 또한 생성된 답변을 마크다운 렌더링 후 `event: done`을 보낼 때 빈 데이터(`data: <span></span>`)를 보내고 있었음.

**해결**

1. `submit_query` 실행 시 `_jobs[job_id]`에 저장하는 튜플을 기존 `(channel, task)`에서 `(channel, task, answer_holder)`로 확장하여 저장.
2. `query_stream`에서 `answer_holder`를 언패킹하여 최종 생성된 답변 문자열(`answer_holder[0]`)을 추출.
3. 추출한 답변을 마크다운으로 렌더링(`md_lib.markdown`)하고, 위키 링크 치환 로직을 적용.
4. 완성된 HTML(answer_html)을 여러 줄에 걸친 SSE 데이터 형식(`data: ` 반복)으로 변환 후 `event: done`과 함께 전송.

**수정 파일**: `wiki_web/routers/query.py`

### 버그 3: "synthesis/ 에 저장" 버튼 클릭 시 LLM이 질문을 재처리하는 현상

**현상**

답변이 완료된 후 "synthesis/ 에 저장" 버튼을 누르면, 완성된 답변이 바로 저장되는 것이 아니라 서버에서 LLM에 동일한 질문을 다시 전송하여 답변을 새로 생성한 뒤 저장하는 비효율적인 방식이 실행됨.

**원인**

`query_result.html` 템플릿의 저장 폼 액션이 동기식 라우터인 `/query/ask`를 가리키고 있었음. 이로 인해 질문 파라미터만 전송되고 기존에 생성된 답변 데이터는 전달되지 않아 서버에서 `run_query`를 처음부터 다시 수행함.

**해결**

1. `query.py` 라우터에 이미 생성된 원문 마크다운(`raw_answer`)을 그대로 받아 저장만 수행하는 `/query/save` POST 엔드포인트를 신규 추가함. (저장 처리 로직은 CLI의 `_save_synthesis` 함수 직접 활용)
2. `query_stream` 라우터에서 `done` 이벤트 발생 시 렌더링하는 `query_result.html` 컨텍스트에 `raw_answer`도 함께 전달하도록 수정.
3. `query_result.html`의 폼 액션을 `/query/save`로 변경하고, `raw_answer`를 hidden `<textarea>` 형태로 전송하도록 수정하여 LLM 재호출 없이 즉시 저장되도록 구조 개선.

**수정 파일**: `wiki_web/routers/query.py`, `wiki_web/templates/partials/query_result.html`

### 개선 1: 위키링크 렌더링 스타일 수정 (글자색 및 배경색 가독성 개선)

**현상**

질문 답변의 결과 화면에서 위키링크 부분(`<em>` 태그)의 글자색과 배경색이 동일하게 렌더링되어 링크된 글자가 보이지 않는 현상 발생.

**원인**

`base.html`에 정의된 `em.wikilink` 클래스의 스타일에서 `color`와 `background` 색상이 유사하게 적용되거나 테마에 따라 동일한 색으로 나타나 가독성을 해침.

**해결**

`em.wikilink` 클래스의 `background` 속성을 `transparent`로 설정하여 배경색을 투명하게 만들고, 글자색은 가시성이 높은 파란색(`color: #1a56db;`)으로 지정함. 또한 강조를 위해 `font-weight: 600`과 밑줄(`text-decoration: underline`) 속성을 추가함.

**수정 파일**: `wiki_web/templates/base.html`

### 개선 2: 질문 답변 진행 과정 SSE 타임라인 UI 도입

**현상**

질문을 입력하고 '답변 생성' 버튼을 누르면 동기식 처리(`/query/ask`)로 인해 10~20초 이상 화면 변화 없이 일방적으로 기다려야 하여 사용자 경험(UX)이 좋지 않음.

**원인**

`query.html`의 폼 액션이 동기식 라우터를 바라보고 있었으며, 기존에 구현된 SSE 스트리밍 라우터(`/query` -> `/{job_id}/stream`)를 템플릿에서 적극적으로 활용하지 않고 있었음.

**해결**

1. `query.html`의 폼 액션을 `/query`(POST)로 변경하여 HTMX 기반 SSE 통신을 시작하도록 전환.
2. `query_progress.html` 템플릿을 전면 개편하여, **4단계 타임라인**(인덱스 읽기 → 페이지 검색 → LLM 생성 → 완료)을 보여주는 UI를 도입.
3. 실시간 경과 시간을 보여주는 타이머와, 현재 진행 중인 단계를 나타내는 스피너 애니메이션(`⟳`) 추가.
4. `query_result.html` 결과 화면을 카드(`query-result-card`) 형태로 감싸, 처리가 완료된 후 진행 패널에서 부드럽게 슬라이드인(Slide-in) 되며 결과가 나타나도록 개선.

### 개선 5: 마크다운 본문 렌더링 개선 및 위키링크 렌더링 로직 분리

**현상**

1. 렌더링된 마크다운 HTML이 기본 CSS(`base.html`)에 연결되지 않아 `h1~h6`, `code`, 테이블 등이 밋밋하게 표시되는 문제가 있었음.
2. `[[Entity]]` 형식의 옵시디언 위키링크가 실제 `.md` 파일(출처)인지 단순 개념 엔티티 태그인지 시각적으로 구별하기 어려웠음.

**해결**

1. **마크다운 CSS 추가**: `base.html`에 `.answer-body` 클래스를 정의하여 마크다운 본문의 줄간격, 헤더 크기, 들여쓰기, 인라인 코드, 테이블 디자인 등을 일관성 있게 렌더링.
2. **렌더링 유틸 분리**: `wiki_web/render.py` 모듈을 신설하여, 질문 응답과 합성 문서 미리보기 양쪽에서 공통으로 마크다운을 렌더링하고 위키링크를 변환하도록 중복 로직 제거.
3. **위키링크 시각적 구분**: 정규식을 통해 `[[파일명.md]]`는 `[🔗 출처]` 형태의 연한 회색 버튼으로, `[[개념어]]`(`.md` 확장자 없음)는 `[🏷 개념어]` 형태의 파란색 알약(Pill) 형태의 엔티티 태그로 구분 렌더링 적용.

**신규 및 수정 파일**: `wiki_web/render.py`, `wiki_web/routers/query.py`, `wiki_web/routers/synthesis.py`, `wiki_web/templates/base.html`

### 개선 6: 마크다운 리스트 파싱 버그 수정

**현상**

LLM이 생성한 응답 중 `**1. 헤더**`와 같은 일반 볼드 텍스트 단락 바로 다음 줄에 `* 항목` 형태의 리스트가 올 경우, Python `markdown` 라이브러리가 빈 줄이 없다는 이유로 전체를 하나의 단락(`<p>`)으로 묶어버려 리스트가 제대로 렌더링되지 않고 한 줄로 이어져 출력되는 문제가 있었음.

**해결**

1. `wiki_web/render.py` 모듈 내에 `_preprocess_md()` 전처리 함수를 추가함.
2. 마크다운 문자열을 줄 단위로 분석하여, 비어있지 않은 줄 바로 다음 줄이 리스트(`*` 또는 `-`)로 시작하는 패턴이 감지되면 두 줄 사이에 강제로 빈 줄(`\n`)을 삽입하도록 수정.
3. 이를 통해 볼드 텍스트 단락과 리스트가 정상적으로 분리되어 각각 `<p>`와 `<ul>`/`<ol>` 태그로 정확히 렌더링되도록 보장함.

**수정 파일**: `wiki_web/render.py`

### 개선 7: 위키 도메인 디렉터리 통합 관리 (Workspace Root) 적용

**현상**

이전 설정에서는 여러 개의 도메인(위키)을 생성할 때 각 도메인마다 완전히 별개의 **위키 루트(절대 경로)**를 개별적으로 지정해야 했음. 이로 인해 도메인들이 분산되고 관리가 번거로움.

**해결**

1. **설정 스키마 마이그레이션 (`wiki_web/config.py`)**:
   - `workspace_root`라는 글로벌 상위 경로 설정을 추가함.
   - 기존의 `wiki_root` 절대 경로 설정을 `workspace_root`와 그 하위의 `folder`(서브 폴더명)로 분리하도록 자동 마이그레이션 로직 구현.
2. **관리 라우터 및 UI 개편 (`wiki_web/routers/admin.py`, `wiki_web/templates/admin.html`)**:
   - `/admin` 페이지 최상단에 **🌐 글로벌 워크스페이스 루트** 섹션을 신설하여 언제든 최상위 폴더를 디렉터리 브라우저로 지정할 수 있게 함.
   - 새 도메인 추가 시 절대 경로를 찾을 필요 없이 **서브 폴더명**만 입력하면 되도록 UI를 간소화함 (`{Workspace Root} / {폴더명}` 형태).
   - 각 도메인의 경로 수정도 서브 폴더명 수정으로 직관성을 향상함.

### 개선 9: 관리 화면(Admin) UI 최적화 및 설정 기능 통합

**현상**

- 도메인 관리(Admin)와 시스템 설정(Settings) 메뉴가 분리되어 있어 관련 설정 시 페이지를 이동해야 하는 불편함.

2. **도메인 상태 뱃지 개선**: 현재 활성화된 도메인이 미초기화 상태(AGENTS.md 없음)일 경우, 오해를 방지하기 위해 "● 활성" 대신 "● 선택됨 (미초기화)"로 라벨을 구체화함.
3. **설정 메뉴 통합**: 기존 `settings.html`의 LLM 모델, API 키, 검색 방식 설정을 `admin.html` 하단에 "시스템 설정" 영역으로 병합하고, 상단 네비게이션(GNB)에서 "설정" 링크를 제거함.

**수정 파일**: `wiki_web/routers/admin.py`, `wiki_web/templates/admin.html`, `wiki_web/templates/base.html`

### 개선 12: 관리 화면 도메인 UI 최적화 및 폴더명 생성 규칙 강화

**현상**

- 도메인 목록에서 아직 초기화되지 않은 기본 도메인도 글로벌 워크스페이스의 전체 절대 경로가 노출되어 화면이 지저분함.
- 도메인 카드의 액션 버튼(이름 변경, 폴더 변경 등) 폼의 상하 여백과 폰트 크기가 불규칙하여 행의 높이가 맞지 않아 어지러움.
- 새 도메인을 추가하거나 기존 도메인 이름을 변경할 때 폴더명을 수동으로 수정해야 하는 "폴더 변경" 폼이 남아있음.
- 도메인 폴더명이 하이픈(`-`) 기반으로 생성되고 있어, 언더스코어(`_`)를 선호하는 사용자 요구와 다름.

**해결**

1. **전체 경로 간소화**: `admin.html`의 도메인 경로 표시 부분(`domain-path`)을 수정하여, 초기화되지 않은 도메인은 절대 경로 대신 짧은 서브 폴더명만 나타나게 처리.
2. **UI 높이 및 정렬 통일**: 활성화, 이름 변경, 위키 초기화, 삭제 버튼 및 입력창의 `padding`과 `font-size`를 통일하여 도메인 카드의 행 높이를 일관성 있게 맞춤.
3. **수동 폴더 변경 폼 제거**: `admin.html`에서 불필요한 "폴더 변경" 폼을 완전히 삭제하고, 해당 라우터(`update-folder`)도 `admin.py`에서 제거함.
4. **폴더명(Slug) 자동 생성 규칙 강화**:
    - `admin.py`의 `add_domain` 및 `rename_domain` 로직을 수정하여 공백 및 특수문자를 언더스코어(`_`)로 치환해 안전한 폴더명으로 변환하도록 정규식을 변경함.
    - 이름 변경 시, 연결된 실제 폴더가 존재한다면 자동으로 새 폴더명으로 변경(`shutil.move()`)하도록 로직을 강화하여 사용자가 폴더명을 직접 신경 쓸 필요를 없앰.

**수정 파일**: `wiki_web/routers/admin.py`, `wiki_web/templates/admin.html`

### 개선 13: OpenWebUI 스타일 기반 디자인 개편 및 위키 초기화 모달(Modal) 적용

**현상**

- 드롭다운 메뉴처럼 보이는 요소가 실제로는 단순 DIV 확장형(`details/summary`)으로 구현되어 주변의 입력 폼 UI들과 디자인 일관성이 떨어지고 공간을 비효율적으로 차지함.
- 디자인 전반이 Pico CSS의 기본 파란색 강조색 등에 머물러 있어, 모던하고 전문적인 느낌(OpenWebUI 스타일)이 부족함.

**해결**

1. **디자인 테마 변경 (OpenWebUI 스타일)**: `base.html`의 CSS `:root` 변수를 커스텀하여, 기본 배경색과 테두리는 미니멀한 화이트/그레이톤으로, 주요 강조색과 버튼은 깊이 있는 무채색(다크/블랙 톤)으로 변경하여 세련되고 일관된 테마를 구축함. (다크모드 대응 포함)
2. **모달(Dialog) 컴포넌트 적용**: 어색했던 `details` 태그 기반의 "위키 초기화" 확장 박스를 제거하고, 버튼 클릭 시 깨끗하고 세련된 오버레이 형태로 등장하는 `HTML <dialog>` 모달 팝업으로 재구현함. 이를 통해 화면 공간을 절약하고 중요한 설정 변경의 UI/UX를 크게 개선함.

**수정 파일**: `wiki_web/templates/base.html`, `wiki_web/templates/admin.html`

### 개선 14: OpenWebUI 스타일 전면 UI/UX 재설계

**현상**

- 기존 레이아웃은 상단 가로 네비게이션(top-nav) 구조로, 화면 폭이 좁아지면 메뉴가 넘치고 도메인 선택기와 겹치는 문제가 있었음.
- Pico CSS 프레임워크 기반이라 커스텀 디자인의 한계가 존재하고, 다크 모드 전환 기능이 없었음.
- 각 페이지의 레이아웃·타이포그래피·간격이 일관되지 않아 전체적인 완성도가 낮았음.

**해결**

1. **레이아웃 전면 교체 (사이드바 구조)**
   - 상단 네비게이션을 완전히 제거하고 220px 고정 너비의 좌측 사이드바로 전환. SVG 아이콘 + 텍스트 조합의 네비게이션 아이템 적용.
   - `app-shell` / `sidebar` / `main-area` / `content-area` 구조로 100vh 전체를 채우는 앱형 레이아웃 구현.
   - 사이드바 하단에 워크스페이스 도메인 선택기와 다크/라이트 테마 토글 배치.

2. **디자인 시스템 재구축 (Pico CSS 제거)**
   - Pico CSS 의존성을 완전히 제거하고 CSS Custom Properties 기반의 자체 디자인 시스템으로 교체.
   - Inter 폰트, 다크 테마 기본값(`--bg: #0f0f0f`), `--radius`, `--border`, `--accent` 등 토큰 체계 정립.
   - `.btn`, `.card`, `.form-input`, `.badge`, `.stat-card` 등 공통 컴포넌트 클래스 전면 재정의.
   - `localStorage`를 통한 테마 영속화 지원.

3. **전 페이지 템플릿 일괄 재작성**
   - `base.html`, `dashboard.html`, `query.html`, `documents.html`, `synthesis.html`, `lint.html`, `settings.html`, `admin.html` 8개 파일 전면 재작성.
   - 모든 페이지에 `.page-header` / `.card` / `.card-header` / `.card-body` 구조 적용.
   - 폼 요소를 `.form-field` / `.form-label` / `.form-input` / `.form-select` / `.form-textarea` / `.form-hint` / `.form-row` 클래스로 통일.
   - `dialog` 모달의 인라인 스타일도 CSS 변수 기반으로 통일하여 다크 모드 대응.

4. **admin.html Jinja2 표현식 IDE 충돌 해소**
   - `onsubmit` confirm 문자열 내 이스케이프 따옴표 제거.
   - 모달 닫기 onclick을 `closeInitModal(id)` 헬퍼 함수로 분리.
   - 인라인 style 내 Jinja2 조건식(`{{ 'block' if ... }}`)을 `data-show` 속성 + JS 초기화 방식으로 교체.

**수정 파일**: `wiki_web/templates/base.html`, `wiki_web/templates/dashboard.html`, `wiki_web/templates/query.html`, `wiki_web/templates/documents.html`, `wiki_web/templates/synthesis.html`, `wiki_web/templates/lint.html`, `wiki_web/templates/settings.html`, `wiki_web/templates/admin.html`

### 개선 15: 문서 청킹 전략 도입 (section / fixed / none)

**배경**

기존 `call_with_file()`은 문서 전체를 단일 LLM 컨텍스트에 넣고 60,000자 초과 시 뒷부분을 잘라냈음. 긴 문서의 후반부 정보가 유실되는 근본적인 한계가 있었음.

**해결**

1. **세 가지 청킹 전략 구현** (`wiki_cli/llm.py`):
   - `section` (기본): 마크다운 헤더(`#`, `##`) 경계로 의미 단위 분리. 구조화된 문서에 적합.
   - `fixed`: 글자 수 기준 고정 길이 분리 + 오버랩. 비구조적 문서나 일반 텍스트에 적합.
   - `none`: 기존 방식 유지 (60,000자 초과 시 트런케이트). 짧은 문서 전용.
2. **청크 상한 제한** (`_cap_chunks()`): 청크 수가 `WIKI_MAX_CHUNKS`(기본 20)를 초과하면 인접 청크를 균등 병합하여 LLM 호출 수를 제한함. `fixed` 전략에서 짧은 청크 크기로 긴 문서를 처리할 때 n+1번 LLM 호출이 발생하는 비용 폭증 방지.
3. **다중 청크 처리 흐름**: 청크별 요약 → 전체 병합 요약(최종 LLM 호출 1회) 방식으로 정보 유실 없이 처리.
4. **설정 UI 추가** (`wiki_web/config.py`, `wiki_web/routers/settings.py`, `wiki_web/templates/settings.html`): 설정 페이지에 "문서 청킹 전략" 카드 추가. `fixed` 선택 시 청크 크기(기본 500자)·오버랩(기본 100자) 입력 필드 동적 표시.

**환경변수**

| 변수 | 기본값 | 설명 |
|---|---|---|
| `WIKI_CHUNK_STRATEGY` | `section` | 청킹 전략 (`none` / `section` / `fixed`) |
| `WIKI_CHUNK_SIZE` | `500` | `fixed` 전략의 청크 크기 (글자 수) |
| `WIKI_CHUNK_OVERLAP` | `100` | `fixed` 전략의 오버랩 (글자 수) |
| `WIKI_MAX_CHUNKS` | `20` | 최대 청크 수 (초과 시 자동 병합) |

**수정 파일**: `wiki_cli/llm.py`, `wiki_web/config.py`, `wiki_web/routers/settings.py`, `wiki_web/templates/settings.html`

### 개선 16: query 페이지 버튼 스타일 일관성 개선

**현상**

`localhost:8000/query` 화면에서 "답변 생성" 버튼(`btn btn-primary`)과 하단의 "synthesis/ 에 저장" 버튼·체크박스가 스타일 체계가 달라 일체감이 없었음. 저장 버튼은 인라인 스타일(`padding`, `font-size`)만 사용하고 있었고, `query_result.html`의 HTML 구조(`<article>/<header>/<footer>`)가 CSS 클래스(`result-header/body/footer`)와 연결되지 않았음.

**해결**

1. **`query_result.html` 구조 수정**: `<article>/<header>/<footer>` 시맨틱 태그를 `<div class="result-header/body/footer">`로 교체하여 `query.html`의 CSS 클래스와 연결.
2. **저장 버튼 통일**: 인라인 스타일 제거, `btn btn-secondary btn-sm` + 저장 SVG 아이콘 적용.
3. **"저장됨" 배지**: `<small>` 태그 → `<span class="badge badge-ok">` + 체크 SVG 아이콘.
4. **자동 저장 체크박스**: `<label>` → `btn btn-ghost btn-sm` 래퍼 + 저장 SVG 아이콘으로 "답변 생성" 버튼과 높이 및 스타일 통일.

### 버그 6: LLM 생성 MD 파일의 이중 YAML frontmatter 문제

**현상**

Ingest로 생성된 entity/concept/source 페이지를 Obsidian에서 열면, 파일 상단에 `created`/`updated`만 있는 YAML frontmatter가 먼저 나타나고, 그 아래에 ` ```yaml ` 코드 펜스로 감싸진 두 번째 YAML 헤더(`title`, `type`, `tags`, `sources` 등)와 본문이 섞여 있어 Obsidian이 메타데이터를 인식하지 못함.

**원인**

`ingest.py`의 `_strip_frontmatter_from_body()` 함수는 본문이 `---`로 시작할 때만 frontmatter를 제거함. 그런데 일부 LLM 모델(gemma4:31b 등)이 YAML을 ` ```yaml\n---\n...\n---\n` 형태의 코드 블록으로 감싸서 출력하면, 이 함수가 패턴을 인식하지 못해 원본 LLM 출력이 그대로 body에 들어감. 이후 `fs.write_page()`가 `{created, updated}`만을 frontmatter로 추가하면서 이중 구조가 만들어짐.

**해결**

`_strip_frontmatter_from_body()` 함수를 `_parse_llm_page(text) → (metadata, body)`로 교체. 세 가지 형식을 모두 처리:
- **형식 A**: `---\nfrontmatter\n---\nbody`
- **형식 B**: ` ```yaml\n---\nfrontmatter\n---\nbody\n``` ` (본문이 코드 블록 안)
- **형식 C**: ` ```yaml\n---\nfrontmatter\n---\n` ``` `\nbody (본문이 코드 블록 밖)

추출한 metadata를 `fs.write_page(path, metadata, body)`에 전달하여 title, type, tags, sources 등이 YAML frontmatter에 올바르게 기록되도록 수정.

기존에 잘못 생성된 wiki MD 파일 71개(entities 55개, concepts 14개, sources 2개)를 일괄 수정.

**수정 파일**: `wiki_cli/ops/ingest.py`, 기존 wiki 파일 71개

### 개선 19: 워크스페이스 구조 분리 — wiki/(Obsidian) + data/(운영 데이터)

**배경**

wiki 폴더를 Obsidian Vault로 연결하려면 마크다운 콘텐츠만 별도 폴더에 격리되어야 함. 기존 구조는 도메인 루트 하위에 `wiki/`(마크다운)와 `raw/`(원본 파일)가 함께 있어 Obsidian Vault 경로로 지정하기 어려웠음.

**해결**

글로벌 워크스페이스 루트 하위에 `wiki/`와 `data/` 두 폴더를 분리하여, 도메인별로 다음 구조로 저장:

```
{workspace_root}/
├── wiki/{domain}/      ← Obsidian Vault 연결 대상 (마크다운 전용)
│   ├── AGENTS.md
│   ├── index.md
│   ├── log.md
│   ├── sources/
│   ├── entities/
│   ├── concepts/
│   └── synthesis/
└── data/{domain}/      ← 운영 데이터 (Obsidian 외부)
    └── raw/            ← 원본 소스 파일 (PDF, 논문 등)
        ├── papers/
        ├── articles/
        └── assets/
```

- `{workspace_root}/wiki/` 폴더 자체를 Obsidian Vault로 등록하면, 도메인별 하위 폴더가 자동으로 탐색됨.
- `AGENTS.md`는 wiki root에 두어 Obsidian에서도 도메인 헌법을 열람 가능.

**주요 변경 사항**

1. **`wiki_web/config.py`**: `get_wiki_root()` 반환 경로를 `ws_root/wiki/{folder}`로 변경. `get_data_root()` 헬퍼 신규 추가 (`ws_root/data/{folder}`).
2. **`wiki_cli/fs.py`**: `wiki_dir(root)` 를 identity 함수(`return root`)로 변경. wiki_root 자체가 wiki 디렉터리이므로 이중 경로 조합 불필요.
3. **`wiki_cli/ops/init.py`**: `run_init(wiki_root, data_root, domain)` 시그니처 변경. wiki 콘텐츠는 `wiki_root`에, `raw/` 디렉터리는 `data_root`에 생성.
4. **`wiki_web/routers/admin.py`**: `_domain_stats`, `init_domain_wiki`, `rename_domain` 함수에서 `wiki/` 및 `data/` 서브디렉터리를 각각 분리 참조. 도메인 이름 변경 시 두 경로 모두 자동 이동(`shutil.move`).
5. **`wiki_web/routers/documents.py`**: `_raw_files`, `upload_file`, `start_ingest`, `_find_source`에서 `data_root` 사용으로 전환. `wiki_root`와 `data_root`를 명확히 분리.
6. **`wiki_web/routers/wiki.py`**: `init_wiki` 최초 설정 엔드포인트에서 `workspace_root` 하위에 `wiki/`, `data/` 경로를 계산 후 `run_init` 호출.

**데이터 마이그레이션**

기존 `/Documents/my-wiki/ai_agent/` 구조:
- `ai_agent/wiki/*` → `wiki/ai_agent/`
- `ai_agent/AGENTS.md` → `wiki/ai_agent/AGENTS.md`
- `ai_agent/raw/` → `data/ai_agent/raw/`

**Obsidian 연결 확인**

`/Documents/my-wiki/wiki` 경로가 Obsidian Vault로 이미 등록되어 있음. 신규 도메인 추가 시 해당 vault 내에 도메인명 폴더가 자동 생성됨.

**추가 조치 (Troubleshooting)**

- **대시보드 500 에러 및 미초기화 표시 현상**: 파일 수정(개선 19)을 완료했음에도 `localhost:8000` 대시보드에서 예전 경로(예: `/Users/yoonani/Documents/my-wiki/ai_agent`)를 읽으려 시도하는 현상 발생.
- **원인 및 해결**: FastAPI(uvicorn) 서버가 `--reload` 없이 백그라운드(`python -m wiki_web`)에서 실행 중이었기 때문에, 코드가 변경되었음에도 메모리에 예전 코드가 캐싱되어 있었음. 백그라운드 프로세스를 강제 종료하고 서버를 재시작(`pkill -f "python -m wiki_web" && python -m wiki_web &`)하여 새 코드를 반영하고 정상 작동(통계 및 경로 인식)을 확인 완료함.

**수정 파일**: `wiki_web/config.py`, `wiki_cli/fs.py`, `wiki_cli/ops/init.py`, `wiki_web/routers/admin.py`, `wiki_web/routers/documents.py`, `wiki_web/routers/wiki.py`
