## 버그 수정

### 버그 1: CLI `wiki init` 명령어 완전 불능 (TypeError)

**현상**

`wiki init` 명령어 실행 시 즉시 `TypeError: run_init() got an unexpected keyword argument 'target'` 발생. CLI로 새 위키를 생성할 수 없음.

**원인**

개선 19(워크스페이스 구조 분리)에서 `run_init` 시그니처가 `(wiki_root, data_root, domain)` 으로 변경되었으나, `wiki_cli/main.py`의 `init` 커맨드는 변경 전 방식인 `run_init(target=target, domain=domain)` 을 그대로 호출하고 있었음. `data_root` 인자 자체가 없고 파라미터 이름도 달라 즉시 TypeError 발생.

**해결**

`main.py`의 `init` 커맨드에서 전달받은 `directory` 경로를 `base`로 삼아 `wiki/`, `data/` 서브 경로를 명시적으로 분리 계산 후 전달하도록 수정.

```python
base = Path(directory).resolve()
wiki_root = base / "wiki"
data_root = base / "data"
run_init(wiki_root=wiki_root, data_root=data_root, domain=domain)
```

**수정 파일**: `wiki_cli/main.py`

### 버그 2: CLI 저장 확인 메시지 미표시 (죽은 코드)

**현상**

CLI에서 `wiki query "..."` 실행 후 답변이 synthesis/ 에 저장되어도 "✓ Saved to synthesis/..." 확인 메시지가 화면에 표시되지 않음.

**원인**

`wiki_cli/ops/query.py`의 `_save_synthesis()` 함수 내 저장 완료 메시지 출력 코드가 `if not True:` 조건 아래 있어 절대 실행되지 않는 죽은 코드 상태였음. 이전 리팩터링 과정에서 분기 조건을 정리하다 실수로 `if not True:` 로 남겨진 것으로 추정.

**해결**

`if not True:` 조건을 제거하고 무조건 출력하도록 단순화. `_save_synthesis()`는 웹 앱에서도 `asyncio.to_thread`로 호출되는데, 웹 컨텍스트에서의 콘솔 출력은 서버 로그로 기록되므로 무해함.

**수정 파일**: `wiki_cli/ops/query.py`

### 버그 3: 위키 초기화 실패 시 500 Internal Server Error

**현상**

관리 화면(`/admin`)에서 위키 초기화 버튼을 눌렀다가 초기화가 실패하면, 에러 메시지 대신 500 Internal Server Error가 발생하여 오류 원인을 알 수 없음.

**원인**

`admin.py`의 `init_domain_wiki` 핸들러가 초기화 실패 시 `admin.html` 템플릿을 직접 렌더링하려 했으나, `admin.html`이 필요로 하는 `workspace_root`, `is_workspace_set`, `model_presets`, `search_tiers`, `is_custom`, `cfg` 등 다수의 컨텍스트 변수를 전달하지 않아 Jinja2 렌더링 단계에서 `UndefinedError` 발생.

**해결**

초기화 실패 시 에러 메시지를 쿼리 파라미터로 포함한 리다이렉트 응답으로 대체. 어드민 페이지의 기존 조회 핸들러(`GET /admin`)가 완전한 컨텍스트를 이미 구성하므로, 리다이렉트 후 자연스럽게 에러 표시 가능. 더 이상 사용되지 않는 `request` 파라미터도 함께 제거.

```python
except Exception as e:
    return RedirectResponse(f"/admin?error={e}", status_code=303)
```

**수정 파일**: `wiki_web/routers/admin.py`

### 버그 4: 관리 화면에서 청킹 설정 저장 안 됨

**현상**

관리 화면(`/admin`) 하단의 시스템 설정에서 "문서 청킹 전략"을 변경하고 저장해도 설정이 반영되지 않고 기본값으로 유지됨.

**원인**

`admin.py`의 `/admin/settings` POST 핸들러가 `chunk_strategy`, `chunk_size`, `chunk_overlap` 세 필드를 처리하지 않아 폼에서 전달된 청킹 설정 값을 무시하고 저장하지 않았음. `settings.py`의 `/settings` POST에는 동일한 처리 로직이 있었으나, 관리 화면으로 설정이 통합(개선 9)된 이후 누락 상태로 방치됨.

**해결**

`/admin/settings` POST 핸들러에 `chunk_strategy`, `chunk_size`, `chunk_overlap` 폼 파라미터를 추가하고, `settings.py`와 동일하게 유효성 보정(`chunk_size` 최솟값 100, `chunk_overlap < chunk_size` 강제) 및 설정 저장 처리 로직을 적용함.

**수정 파일**: `wiki_web/routers/admin.py`

### 버그 5: 동기식 질문 답변(`/query/ask`)에서 저장 기능 불능

**현상**

`/query/ask` (동기식) 엔드포인트로 질문 답변을 받은 후 "synthesis/ 에 저장" 버튼을 누르면 저장이 되지 않거나, 빈 내용으로 저장됨.

**원인**

`ask_sync` 핸들러가 `query_result.html` 템플릿을 렌더링할 때 `raw_answer` 컨텍스트 변수를 전달하지 않았음. `query_result.html`은 저장 폼의 hidden 필드(`<textarea name="raw_answer">`)에 이 값을 채우는데, 값이 없으면 `POST /query/save` 요청 시 빈 문자열이 전달됨. 또한 예외 발생 시 `answer` 변수가 정의되지 않은 상태에서 참조하는 문제도 잠재적으로 존재했음.

**해결**

`try` 블록 진입 전에 `raw_answer = ""` 로 초기화하고, `run_query()` 반환값을 `raw_answer`로 받은 뒤 템플릿 컨텍스트에 명시적으로 전달하도록 수정.

**수정 파일**: `wiki_web/routers/query.py`

### 버그 6: 도메인 폴더 중복 시 하이픈/언더스코어 혼합 (네이밍 불일치)

**현상**

도메인 추가 시 동일한 이름을 입력하면 폴더명 충돌 방지 로직이 `my_wiki-1` 처럼 언더스코어와 하이픈이 혼합된 폴더명을 생성함.

**해결**

중복 방지 카운터를 `f"{slug}_{counter}"` 로 변경하여 언더스코어 규칙으로 통일.

### 버그 7: Obsidian 위키링크가 기존 페이지를 찾지 못하고 새 노트를 생성하는 문제

**현상**

Ingest로 생성된 entity/concept 페이지들이 Obsidian vault에 있음에도 불구하고, `[[Entity Name]]` 형태의 위키링크를 클릭하면 기존 파일로 이동하지 않고 새 노트를 생성함.

**원인 (3가지 복합)**

1. **파일명 불일치**: entity/concept 페이지가 `google-agentspace.md` (slug 기반)로 저장되는 반면, 생성되는 링크는 `[[Google Agentspace]]` 형태. Obsidian은 파일명과 alias 두 가지로 링크를 해결하는데, 파일명이 다르면 alias에 의존해야 함.

2. **alias 미설정 또는 불일치**: `fs.write_page()`는 `_meta.get("title")`이 있을 때만 alias를 추가함. `_parse_llm_page()` 파싱이 실패하거나 LLM이 prompt의 `title: <human-readable display name>` 지시를 자유롭게 해석해 다른 이름을 쓰면, alias가 없거나 링크 텍스트와 다른 alias가 설정됨. 결과적으로 `[[Google Agentspace]]`를 해결할 alias가 없어 새 노트 생성.

3. **링크 이름 선작성 문제 (순서 문제)**: ingest 흐름에서 source 페이지(step 2)가 entity/concept 페이지(step 4)보다 먼저 작성됨. source 페이지 작성 시 LLM이 entity 이름을 자유롭게 쓰고, entity 페이지 작성 시 LLM이 또 다른 이름을 결정하면 링크 텍스트와 파일명/alias가 달라짐.

**해결**

세 가지 원인을 모두 제거:

1. **entity/concept 파일명을 display_name 기반으로 변경** (`_write_or_update_page`):
   - 기존: `kind/{slug}.md` (e.g., `entities/google-agentspace.md`)
   - 변경: `kind/{display_name}.md` (e.g., `entities/Google Agentspace.md`)
   - Obsidian에서 `[[Google Agentspace]]`가 파일명으로 직접 해결됨. alias 불필요.

2. **title 강제 설정** (`_write_or_update_page`):
   - LLM 응답 파싱 후 `_meta["title"] = display_name`으로 덮어써 title이 항상 display_name과 일치하도록 보장.
   - `fs.write_page()`가 이 title로 `aliases: [display_name]`를 추가 → 파일명·alias·링크 3중 일치.

3. **entity/concept 계획을 source 페이지 작성 전으로 이동** (ingest 순서 변경):
   - 기존 순서: overview → source page → entity/concept pages
   - 변경 순서: overview → **plan entities/concepts** → source page (with exact names) → entity/concept pages
   - `_plan_related_pages()` 신규 함수: entity/concept slug 목록만 먼저 추출
   - `_page_display_name()` 신규 함수: slug → display_name 변환 단일 책임
   - source 페이지 프롬프트에 확정된 entity/concept display_name 목록을 명시해 LLM이 정확한 이름으로 링크를 작성하도록 강제

**마이그레이션 호환**: 기존 slug-based 파일이 존재할 경우 legacy_path 폴백으로 본문을 읽어 업데이트 내용을 보존함.

**수정 파일**: `wiki_cli/ops/ingest.py`

## 성능 개선 상세

### 개선 1: ingest update 증분 병합 (7.4)

**변경 전**

`_write_or_update_page()`의 update 경로는 전체 `overview`(소스 전체 분석)와 `existing_body` 전체를 프롬프트에 넣고 LLM에게 페이지 전체를 재생성하도록 요청했음.

**변경 후**

세 단계로 분리:

1. `_extract_relevant_evidence(overview, display_name)`: overview에서 해당 엔티티/개념 관련 줄과 전후 문맥만 필터링. 관련 내용이 충분하지 않으면 전체 overview 반환.

2. delta 프롬프트: `existing_body`는 컨텍스트로만 전달하고 LLM은 추가할 내용만 아래 형식으로 출력하도록 요청:
   ```
   SECTION: <기존 섹션 이름>
   <추가할 불렛/문장>

   NEW_SECTION: <새 섹션 이름>
   <새 섹션 전체 내용>
   ```
   추가할 내용이 없으면 `NO_UPDATE` 출력 → 조기 반환.

3. `_apply_delta(body, delta)`: `SECTION:` 지시는 기존 섹션 말미에 append, `NEW_SECTION:`은 body 끝에 추가. frontmatter의 `sources` 목록은 코드에서 직접 갱신.

**효과**: 입력 토큰(관련 evidence만 전달)과 출력 토큰(delta만 출력) 모두 절감. 기존 내용 보존 보장.

---

### 개선 2: 청크 단계 evidence 추출 구조화 (7.6)

**변경 전**

`_chunk_and_call()`은 각 청크에 원본 `summary_prompt` 전체를 그대로 전달해 청크마다 완성형 분석을 생성한 뒤, 마지막에 모든 청크 결과를 다시 통합 요약했음 (N+1회 완성형 출력).

**변경 후**

청크 단계와 통합 단계를 역할 분리:

1. 청크별 호출: 원본 프롬프트 대신 경량 evidence 추출 지시 사용 (`max_tokens=1024`). "엔티티, 개념, 핵심 주장·수치를 간결한 목록으로 추출"만 수행.

2. 최종 통합 호출: 원본 `summary_prompt`에 모든 청크의 evidence 목록을 붙여 1회 완성형 분석 수행.

**효과**: 청크당 출력 토큰이 기존 완성형 요약 대비 대폭 감소. 통합 해석 품질은 최종 1회 호출에 집중.
