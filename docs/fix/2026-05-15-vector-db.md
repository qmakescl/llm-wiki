# 2026-05-15 Vector DB 구현 기록

---

## 배경

`docs/vector_db_implementation_guidelines.md`는 기존 query-time lazy embedding 검색을 ingest-time chunk vector indexing 구조로 확장하는 방향을 제안했다.

기존 `embedding` 검색은 query 시점에 markdown 파일을 훑고 각 파일 앞부분만 임베딩했다. 이 방식은 긴 문서나 여러 주제가 섞인 문서에서 관련 근거 section을 정확히 찾기 어렵다.

이번 작업에서는 외부 서버 없이 로컬 SQLite 파일로 동작하는 1차 vector index를 구현했다.

---

## 구현 내용

### 개선 1: SQLite 기반 Vector Index 모듈 추가

**변경 내용**

- `wiki_cli/vector_index.py` 신규 추가
- `<wiki_dir>/.vectors/vector_index.sqlite`에 chunk metadata와 embedding 저장
- `refresh_page`, `refresh_all`, `search_chunks`, `delete_page`, `stats`, `clear` 제공
- embedding model은 기존과 같은 `sentence-transformers` `all-MiniLM-L6-v2` 사용
- `sqlite-vec` 같은 확장 없이 SQLite + JSON vector + Python cosine similarity로 시작

**주요 스키마**

| 필드 | 내용 |
|---|---|
| `chunk_id` | wiki path, heading, chunk index, content hash 기반 안정 ID |
| `wiki_path` | wiki root 기준 상대 경로 |
| `page_title` | frontmatter title 또는 파일 stem |
| `heading` | chunk가 속한 heading |
| `chunk_index` | 페이지 내 chunk 순번 |
| `chunk_text` | LLM context에 직접 전달할 chunk text |
| `content_hash` | chunk text SHA1 |
| `model_name` | embedding model name |
| `embedding_dim` | vector dimension |
| `embedding` | JSON float list |
| `created_at`, `updated_at` | UTC timestamp |

### 개선 2: 검색용 Markdown Chunking 추가

**변경 내용**

- `chunk_markdown_for_search()` 추가
- `WIKI_CHUNK_STRATEGY`, `WIKI_CHUNK_SIZE`, `WIKI_CHUNK_OVERLAP` 재사용
- `section` 전략은 heading 경계를 우선 보존
- 긴 section은 fixed chunk + overlap으로 추가 분할
- `fixed` 전략은 문자 기준 overlap 적용
- `none` 전략에서 긴 문서는 앞부분만 쓰지 않고 fixed fallback 사용

---

### 개선 3: Ingest 후 Vector Upsert 연결

**변경 내용**

- source summary page 작성 직후 vector index refresh 수행
- entity/concept page 생성 또는 업데이트 후 vector index refresh 수행
- vector indexing 실패가 ingest 실패로 이어지지 않도록 best-effort 처리
- `sentence-transformers` 미설치 또는 DB 오류 시 warning만 남기고 기존 ingest 흐름 유지

---

### 개선 4: `WIKI_SEARCH=vector` 검색 티어 추가

**변경 내용**

- `wiki_cli/search.py`에 `vector` tier 추가
- 기존 `SearchResult`에 optional `metadata` 추가
- vector 검색 결과는 기존 `SearchResult` 형태를 유지하되 chunk metadata를 포함
- vector DB가 비어 있거나 모델/DB 오류가 있으면 grep fallback 수행
- 기존 `grep`, `bm25`, `embedding` tier는 유지

**Vector result metadata**

```python
{
    "kind": "vector_chunk",
    "heading": "...",
    "chunk_text": "...",
    "chunk_id": "...",
    "page_title": "...",
    "chunk_index": 0,
}
```

---

### 개선 5: Query Context Builder 개선

**변경 내용**

- `_build_context()`가 `metadata.kind == "vector_chunk"`인 결과를 인식
- vector 검색에서는 page 전체를 다시 펼치지 않고 관련 chunk text를 LLM context에 직접 전달
- 기존 검색 결과는 기존처럼 search index entry 또는 원문 page fallback을 사용

---

### 개선 6: Web 설정과 CLI 관리 명령 추가

**Web 설정**

- `wiki_web/config.py`의 `SEARCH_TIERS`에 `vector` 추가
- 설정 저장 방식은 기존 `search_tier` / `WIKI_SEARCH` 흐름 그대로 사용

**CLI 명령**

```bash
wiki vector rebuild
wiki vector stats
wiki vector clear
```

| 명령 | 내용 |
|---|---|
| `wiki vector rebuild` | 모든 wiki page를 다시 chunk vector index에 반영 |
| `wiki vector stats` | vector DB 경로, page 수, chunk 수 출력 |
| `wiki vector clear` | 로컬 vector DB 삭제 |

## 동작 정책

- `vector` tier는 명시적으로 `WIKI_SEARCH=vector`일 때만 사용한다.
- 최초 배포 시 자동 전체 indexing은 강제하지 않는다.
- 새로 ingest되거나 갱신된 page는 자동으로 vector index에 반영한다.
- 기존 markdown page 전체를 반영하려면 `wiki vector rebuild`를 실행한다.
- vector DB가 비어 있거나 embedding model/DB를 사용할 수 없으면 grep으로 fallback한다.
- 기존 `.embeddings/` pickle cache는 그대로 유지하며, 새 `.vectors/` DB와 혼동하지 않는다.

---

## 남은 한계

- page 삭제 이벤트를 자동 감지하는 파일 watcher는 아직 없다. 삭제된 page는 `delete_page()` 또는 전체 rebuild/clear 정책으로 정리해야 한다.
- 모델명 또는 chunk 설정 변경 감지는 아직 자동화하지 않았다. 설정 변경 후에는 `wiki vector rebuild` 실행이 필요하다.
- 현재는 SQLite 확장 없이 brute-force cosine search를 사용한다. 개인 위키 규모에는 단순하고 안정적이지만, 대규모 문서에서는 `sqlite-vec` 같은 확장 도입을 검토할 수 있다.
