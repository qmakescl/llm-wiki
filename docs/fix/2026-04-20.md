### 버그 1: HTMX 타깃 div 조건부 렌더링으로 업로드 후 화면 멈춤

**현상**

**해결**

`#file-list` div를 항상 렌더링하고, 빈 상태 메시지를 내부에 배치.

**수정 파일**: `wiki_web/templates/documents.html`

### 버그 2: asyncio.get_event_loop() 오용

**현상**

Python 3.10+ 환경에서 deprecation 경고 또는 잘못된 루프 반환.

**원인**

`ProgressChannel.__init__`에서 `asyncio.get_event_loop()` 사용. `ProgressChannel`은 항상 async 핸들러 내부에서 생성되므로 실행 중인 루프를 직접 가져와야 함.

**해결**

```python
# 수정 전
self._loop = asyncio.get_event_loop()

# 수정 후
self._loop = asyncio.get_running_loop()
```

**수정 파일**: `wiki_web/progress.py`

### 버그 3: htmx-indicator에 display:none 인라인 스타일

**현상**

업로드/Ingest/질문 중 로딩 스피너가 표시되지 않음.

**원인**

**해결**

인라인 `display:none` 제거, class만 유지.

**수정 파일**: `wiki_web/templates/documents.html`, `partials/file_row.html`, `query.html`, `lint.html`

### 버그 4: CLI에서 Ollama 설정 파일 미반영

**현상**

웹 UI에서 `ollama_base_url`을 원격 IP로 설정해도 CLI(`wiki` 명령어)는 `localhost:11434`를 사용.

**원인**

`wiki_cli/main.py`의 `cli()` 그룹이 `~/.config/llm-wiki/config.json`을 전혀 로드하지 않음.

**해결**

`cli()` 그룹 시작 시 설정 파일을 읽어 환경변수에 적용.

```python
def _apply_saved_config() -> None:
    try:
        from wiki_web import config as _cfg
        _cfg.apply_env(_cfg.load())
    except Exception:
        pass
```

**수정 파일**: `wiki_cli/main.py`

### 버그 5: litellm provider prefix 누락으로 Ingest 오류

**현상**

```
litellm.BadRequestError: LLM Provider NOT provided.
Pass in the LLM provider you are trying to call. You passed model=gemma4:31b
```

**원인**

설정 파일에 `"model": "gemma4:31b"` 형식으로 저장 시 litellm이 provider를 알 수 없음. litellm은 `ollama/gemma4:31b` 형식의 provider prefix를 요구.

**해결**

`resolve_model()`에서 `name:tag` 형식(슬래시 없고 콜론 있음)을 자동으로 `ollama/` prefix 추가. 클라우드 모델(`gpt-4o`, `claude-...`)은 콜론을 사용하지 않아 충돌 없음.

```python
def resolve_model(override: str | None) -> str:
    m = override or default_model()
    if ":" in m and "/" not in m:
        m = f"ollama/{m}"
    return m
```

**수정 파일**: `wiki_cli/llm.py`

### 버그 6: Ingest 시 소스 PDF를 LLM에 이중 전송

**현상**

단일 문서 Ingest에 과도하게 긴 시간 소요.

**원인**

Step 1(분석)에서 PDF 전체를 LLM에 전송하고, Step 2(요약 페이지 생성)에서 `call_with_file()`로 동일한 PDF를 다시 전송. Step 2의 프롬프트에는 Step 1의 분석 결과(`overview`)가 이미 포함되어 있으므로 재전송 불필요.

**해결**

Step 2를 `call_with_file()` → `call()`로 교체. `overview`만으로 충분.

**수정 파일**: `wiki_cli/ops/ingest.py`

### 개선 4: 워크스페이스 구조 분리 — `wiki/`(Obsidian) + `data/`(운영 데이터)

**배경**

wiki 폴더를 Obsidian Vault로 연결하려면 마크다운 콘텐츠만 별도 폴더에 격리되어야 함. 기존 구조는 도메인 루트 하위에 `wiki/`(마크다운)와 `raw/`(원본 파일)가 함께 있어 Obsidian Vault 경로로 지정하기 어려웠음.

**새 디렉터리 구조**

```
{workspace_root}/
├── wiki/{domain}/      ← Obsidian Vault 연결 대상 (마크다운 전용)
│   ├── AGENTS.md
│   ├── index.md, log.md
│   ├── sources/, entities/, concepts/, synthesis/
└── data/{domain}/      ← 운영 데이터 (Obsidian 외부)
    └── raw/            ← 원본 소스 파일 (PDF 등)
```

**수정 파일**: `wiki_web/config.py`, `wiki_cli/fs.py`, `wiki_cli/ops/init.py`, `wiki_web/routers/admin.py`, `wiki_web/routers/documents.py`, `wiki_web/routers/wiki.py`

### 버그 7: LLM 생성 MD 파일의 이중 YAML frontmatter

**현상**

Ingest로 생성된 파일을 Obsidian에서 열면 상단에 `created`/`updated`만 있는 YAML이 먼저 나오고, 그 아래에 ` ```yaml ` 코드 펜스로 감싸진 두 번째 YAML 헤더와 본문이 섞여 있음.

**원인**

`_strip_frontmatter_from_body()`가 `---` 시작 패턴만 처리. 일부 LLM이 ` ```yaml\n---\n...\n---\n` 형태로 출력하면 처리하지 못하고 raw 출력이 body에 그대로 들어감. 이후 `fs.write_page()`가 `{created, updated}`만 frontmatter로 추가해 이중 구조 발생.

**해결**

`_strip_frontmatter_from_body()` → `_parse_llm_page(text) → (metadata, body)` 교체. 세 가지 출력 형식(bare `---`, 코드 블록 내 본문, 코드 블록 외 본문) 모두 처리. 기존 wiki 파일 71개 일괄 수정.

**수정 파일**: `wiki_cli/ops/ingest.py`, 기존 wiki MD 파일 71개

### 버그 8: Admin 페이지 커스텀 모델 입력창 미표시

**현상**

`/admin`에서 LLM 모델 셀렉트를 "직접 입력..."으로 선택해도 커스텀 모델 ID 입력창이 나타나지 않음. 이미 커스텀 모델(`gemma4:31b` 등)이 설정된 상태로 페이지를 열어도 입력창이 보이지 않음.

**원인**

`#admin-custom-wrap` 요소를 보이게 하는 JavaScript IIFE가 해당 DOM 요소보다 **앞에** 위치한 `<script>` 블록 안에 있어, 실행 시점에 `getElementById('admin-custom-wrap')`이 `null`을 반환.

**해결**

IIFE를 `document.addEventListener('DOMContentLoaded', ...)` 으로 교체하여 DOM 파싱 완료 후 실행되도록 수정.

**수정 파일**: `wiki_web/templates/admin.html`

### 개선 5: Obsidian 위키링크 연결 — `aliases` frontmatter 추가

**현상**

생성된 MD 파일에서 `[[Google Agentspace]]` 링크가 Obsidian에서 연결되지 않음. 파일명은 `google-agentspace.md`(slug)인데 링크는 display name이어서 Obsidian이 파일을 찾지 못함.

**해결**

- 각 파일 YAML frontmatter에 `aliases: [title]` 추가. Obsidian이 `[[Google Agentspace]]` → `google-agentspace.md` 자동 연결.
- `fs.write_page()`에 `title` 있을 때 `aliases` 자동 추가 로직 삽입 (이후 생성 파일도 적용).
- 기존 wiki 파일 72개 일괄 수정.

**수정 파일**: `wiki_cli/fs.py`, 기존 wiki MD 파일 72개

### 개선 6: Ingest 프롬프트 강화 및 title 추출 버그 수정

**문제점**

1. entity/concept 페이지 생성 프롬프트에 YAML 필드 명세가 없어 `title`/`type`/`tags`/`sources` 누락 위험.
2. source 페이지 title을 `_extract_title(raw_text)` (raw LLM 출력 파싱)로 추출 → 코드 블록 안 title 누락 가능.
3. entity/concept 페이지 index title을 `slug.replace("-"," ").title()` 추정값으로 사용 → 실제 title과 불일치.
4. 시스템 프롬프트가 `[[PageName]]`의 PageName이 slug인지 display name인지 불명확 → LLM이 slug 형태 링크 생성 위험.

**해결**

- `_SYSTEM` 프롬프트: display name 사용, slug 금지, code fence 금지 명시.
- source/entity/concept 프롬프트: bare `---` 블록 형식 + `title/type/tags/sources` 필드 명세 포함.
- title 추출: `_parse_llm_page()` 결과 `_meta.get("title")` 우선 사용.

**수정 파일**: `wiki_cli/ops/ingest.py`
